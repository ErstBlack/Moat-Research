Recommendation: **defer-with-rationale** until n>15 or decision-splitting conflict emerges. Trigger conditions codified in docs/calibration/2026-05-05-n8-pass.md.
- **Composite score validation:** The somd-cameras rescore against RUBRIC.md produced 7.221, confirming the scoring formula and weights work as intended for a known-good moat. No rubric adjustments needed yet; revisit after ≥3 more lane-1 candidates are scored.
- **Robots.txt ambiguity resolution:** When an API endpoint's robots.txt contains `Disallow: /` (appears to block all programmatic access), prefer bulk-download paths that are explicitly documented and clearly allowed over live API endpoints. Applied to USGS/NWS fusion: selected NCEI bulk CSV over api.weather.gov.
- **Multi-operator coverage claims must be per-operator verified:** Government program pages (e.g., DDOT's DC shared-mobility operator list) often advertise N operators with public feeds, but the operators churn faster than the page updates. The 2026-05-04 DC GBFS recheck found only 1/5 listed operators (Capital Bikeshare) reachable and unrestricted; Lime had Disallow:/, Helbiz was defunct (504), Spin had retired the DC feed (404), Lyft's standalone S3 path was stale since 2023. Treat any "fusion across all operators in jurisdiction X" thesis as unverified until each operator endpoint is independently confirmed live + within hard constraints, and adjust the moat thesis honestly when verification narrows the scope.
- **S3-hosted GBFS robots.txt 403 ≠ Disallow:** S3 buckets serving GBFS data (e.g., gbfs.lyft.com, gbfs.capitalbikeshare.com) return HTTP 403 AccessDenied on `/robots.txt` because no robots.txt file is configured on the bucket. This is "robots.txt absent" (no policy = no restriction) and not a Disallow. Do not treat S3 403-on-robots as a hard-constraint hit; treat it as the absence-of-policy case in CONSTRAINTS §2.
- **GBFS feed UA sensitivity:** Some GBFS hosts (notably gbfs.lyft.com) appear to gate responses on User-Agent. The 2026-05-04 prior pass got 403 from gbfs.lyft.com; the recheck got 200 across all six core feeds with `User-Agent: moat-research/0.1 (research; contact: ...)`. When a GBFS host returns 403, retry with a descriptive UA before concluding it's blocked.
- **Cross-source-fusion (Lane 3) thesis stress test:** A Lane-3 brief that claims "the time-aligned join is the moat because the underlying sources don't preserve millisecond-aligned join keys" is almost always wrong if both raw sources timestamp their records. If both inputs are in public archives with timestamps, any analyst can reconstruct the historical join from public data at any time, so the join itself is not defensible. The defensible artifact is the *derived features* (correlation tables, distributions, embeddings) computed from the join — that's Lane 4 (derived-artifact, compute-as-barrier), not Lane 3. Apply this test to every cross-source brief before scoring: if the historical raw-source join is reconstructible by anyone with the public archives, the brief belongs in Lane 4 or should be dismissed. Confirmed 2026-05-04 against `coops_ais_coastal_fusion`: CO-OPS API + NOAA AIS bulk archive both have public timestamps and historical access, so the original Lane-3 thesis collapses.
- **NDBC publishes its own complete spectral archive (Lane-1 trap):** NDBC realtime2 spectral wave density files (`<station>.spec`, `<station>w.txt`) are overwritten on each observation, BUT NDBC itself maintains a complete public archive at the same :10/:40 sub-hourly cadence: `/data/swden/{Jan|Feb|...|Dec}/<station><MM><YYYY>.txt.gz` (monthly intermediate) and `/data/historical/swden/<station>w<YYYY>.txt.gz` (annual rollup). Both are open HTTP, public domain, identical column schema (YY MM DD hh mm + frequency-bin energies). Confirmed 2026-05-04 against 41001/41002 across 2024/2025/2026 files. The "10–50 minute realtime cadence" wishlist framing was speculative; real cadence is twice-hourly at fixed :10/:40 minutes, fully preserved in the public archive. Lesson for any "ephemeral NOAA observation file" brief: ALWAYS check `/data/historical/<product>/` and `/data/<product>/<Mon>/` paths before asserting Lane 1. NOAA's preservation pipeline (realtime → monthly → annual) is the default, not the exception.
- **CONSTRAINTS §5 verification methodology:** For any Lane-1 brief, the disqualifier check is: (1) read upstream's docs page for "historical data" / "archive" links; (2) look for both annual rollup and intermediate (monthly/daily) public paths; (3) download a sample file from the historical archive and inspect column headers — if it preserves the same `mm` (minute) field as the realtime feed at the cadence the brief depends on, the moat collapses; (4) check `Last-Modified` headers on the historical files to assess archive freshness lag. The brief's actual cadence requirement, not its speculative wishlist framing, is what the archive must close to trigger §5. Worked cleanly against ndbc_realtime_buoys 2026-05-04 (auto-rejected) and is now the standard pre-promotion check for Lane-1 candidates.
- **Lane 4 entity-resolution as defensibility:** When a Lane-4 brief depends on raw PDFs/documents that ARE publicly archived (so the raw passes through §5 unscathed only as inputs, not as the moat), the brief's defensibility narrative MUST rest on three pillars or it collapses to a one-shot reproducible analyst job: (1) compute-as-barrier — OCR + NER + structured extraction at scale that an analyst would need months to reproduce; (2) ongoing-update compounding — weekly/monthly cadence so the corpus's current value depends on continuous capture, not a one-time pass; (3) cross-jurisdiction or cross-source entity resolution — canonicalizing the same entity (contractor, physician, vessel, parcel) across N publishers where each source uses different identifiers. If a brief can't articulate at least 2 of those 3, it isn't really Lane 4 and won't survive operator scrutiny. Apply during discovery synthesis as a pre-promotion check on Lane-4 candidates, parallel to the Lane-1 §5 archive-completeness check.

- **Free-API-token gating ≠ auth-bypass:** Many transit agencies and government APIs require a free developer token (registration form, no payment, immediate issuance). This is normal authorized use, NOT CONSTRAINTS §1 (auth-bypass / CFAA-adjacent). What §1 actually targets is using credentials you don't have permission to hold (stolen, bypassed, or session-token abuse). The brief-level concern is the developer agreement's redistribution / archival clauses, which are §2 (ToS) territory — many transit dev agreements permit analytical use but forbid public re-sharing of feed contents, which is compatible with this project's private-corpus moat. Confirmed during 2026-05-04 GTFS-RT discovery pass: AC Transit returned 401 without a token, but the token landscape is normal and per-agency. Treat free-token agencies as in-scope for Lane 1, with brief-stage per-agency ToS audit.

- **System-of-record archival kills Lane-1 across whole agency types:** When an agency's primary data product IS a long-term archival platform (e.g., Legistar for municipal council records, court-records portals, Socrata-backed open-data sites), Lane 1 universally fails §5 for that agency type because preservation is the platform's purpose. During discovery synthesis, dismiss the entire category at first-pass rather than per-instance: city council legislative platforms, court-records public-access portals, Socrata/Tyler/CKAN-backed open-data sites are §5-killing categories. Re-route any apparent moat to Lane 4 (compute-as-barrier on the archived corpus) and apply the Lane-4 entity-resolution test above. Confirmed 2026-05-04 against NYC Council Legistar, NJ Judiciary public-access portal, Chicago Open Data 311.

- **AIS bulk-archive properties (NOAA MarineCadastre):** AIS data via marinecadastre.gov is bulk-only, hosted at coast.noaa.gov/htdata/CMSP/AISDataHandler/{year}/, with a 6–12 month publishing latency (2026 archive did not exist yet on 2026-05-04). Format changed from daily .zip in 2024 → daily .csv.zst (zstd) starting 2025; full year ≈ 81.5 GB. Public domain. Any brief touching AIS must NOT assume real-time or near-real-time; AISHub (live AIS) requires a data-sharing agreement and is not publicly accessible.

- **Lane 2 "capture-before-the-door-closes" conditional-moat reasoning:** Distinct from Lane 1 ephemeral-data logic. The moat is conditional on (i) an off-platform durable archive that outlasts the political/regulatory vulnerability AND (ii) verified evidence that the public source has been withdrawn/restricted before. A Lane-2 brief must not duplicate a currently-public archive (that's §5 reconstructibility, not Lane 2); instead it must justify why capturing now prevents future loss. Applied 2026-05-05 to USDA APHIS Animal Welfare (2017 ACIS withdrawal precedent + HSUS litigation history = demonstrated political risk). Revisit if administrations change or source restoration occurs.

- **Free public aggregators collapse §5:** When verifying whether raw data can be reconstructed, include free aggregators (e.g., Dolthub's hospital-price-transparency-v3 repo) in the reconstructibility check. A brief claiming to aggregate structured hospital-pricing data from CMS fails §5 not because CMS archives it, but because Dolthub publishes the same aggregation for free. Confirmed 2026-05-05 during discovery synthesis: rejected Hospital Price Transparency entry on this basis. Different from the paid-incumbent precedent (Interline, S&P Velocity Suite) which don't trigger §5 because commercial barriers don't count as public reconstructibility.

  - **Lane 1 (ephemeral):** Defensibility=0 if the publisher or any third party archives the source at the brief's required cadence. Verification: check /data/historical/, /data/<product>/<month>/, annual rollups on upstream. If found with same-cadence preservation, brief fails CONSTRAINTS §5 → auto-reject.   - **Lane 2 (soon-to-be-restricted):** Defensibility rests on (i) off-platform durable archive AND (ii) demonstrated historical precedent of source restriction/removal (USDA APHIS 2017 ACIS withdrawal, HSUS v. - **Lane-2 detail-layer carve-out (2026-05-05 T2):** When a candidate's bulk-data dump is publicly archived (data.gov mirrors, agency-side bulk download paths, CKAN/Socrata layers), the basic "duplicate the bulk dump" framing fails CONSTRAINTS §5 — but the Lane-2 moat can still survive if the brief carves out a specific detail layer that is (a) NOT in the bulk dump (case-narrative fields, region-level disposition memos, full inspection PDFs vs. summary tables, e-filing case-tracker records) AND (b) demonstrably subject to political-restriction precedent at the discoverability layer (e.g., 2017 EPA OECA homepage scrubs targeted aggregate dashboards, not bulk dumps; 2017 Robb-era NLRB GC publication-restriction memos targeted regional case detail, not Board decisions). - **Lane 3 survival condition formalized (2026-05-05 T2 → resolved T2 of 20260505T125333Z-88ac55):** Seven candidate Lane-3 fusions evaluated across four discovery passes; zero survived. All failures shared the same root cause: both inputs were publicly archived with timestamps, making the historical join reconstructible by any analyst (CONSTRAINTS §5 → auto-reject).   **3-step polite-alternate-path checklist** (apply before dismissing any Akamai-gated federal endpoint):
  1. **data.gov bulk mirror** — search `catalog.data.gov/dataset?q=<agency>+<topic>` for a bulk download dataset published by the same agency. US government data is frequently mirrored at data.gov even when the agency's Drupal front-end is Akamai-gated.
  2. **FOIA reading room** — check `<agency-domain>/foia` or `<agency-domain>/about/foia-reading-room` for proactively-released datasets. Many agencies publish bulk enforcement records under FOIA E-FOIA proactive disclosure, bypassing the UI entirely.
  3. **Direct subdomain / alternate domain** — if the main domain is gated, try individual program-office subdomains, `api.<agency>.gov`, or the agency's OIG/IG site which may host the same data without the Akamai layer. For multi-regulator portals (FFIEC), try each member agency individually (OCC, FDIC, Fed, NCUA, CFPB).
